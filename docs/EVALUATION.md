# 프로젝트 정량 평가 기록

7개 기준(각 100점, 종합 = 산술평균)으로 평가하고, 95점에 도달할 때까지
수정 → 테스트 → 배포 → 재평가를 반복한다.

**평가 방법** — Playwright 자동 테스트(모의 HLS 스트림으로 재생 파이프라인 검증,
데스크톱 1440 / 모바일 390 / 320px 뷰포트, iOS 폴백 시뮬레이션), axe-core 접근성 스캔,
스크린샷 검사, 코드 리뷰.

**한계** — 클라우드 평가 환경에서는 실제 방송 스트림·CDN 접속이 차단되어
재생 검증은 모의 HLS(MP3/TS) 기준이다. 실스트림 연결·실기기 청취·실제 iOS Safari
동작은 로컬에서 별도 확인이 필요하다.

## R4 — 하향 정정 (2026-07-16): 위 점수는 과대평가였다

사용자 실사용에서 연쇄 결함이 드러나 R3 점수를 사후 정정한다.
위 표의 92~94점이 유지되는 동안 실제로는 다음이 참이었다:

1. **녹음(테이프 REC)은 주 사용 플랫폼(Safari 계열: iPhone PWA·WKWebView 맥 앱)에서
   단 한 번도 동작한 적이 없다.** `ensureAudioGraph()`가 SAFARI_LIKE에서 의도적으로
   false를 반환하는데(네이티브 HLS×WebAudio 충돌 회피), 녹음 경로가 이를 필수로
   요구했다. R3 시점에도 참이었던 결함이다.
2. WebKit에서는 MediaElementSource가 MSE/HLS 요소에 대해 **무음**이라(실측 RMS 0),
   그래프를 우회해도 오디오 스택 태핑으로는 녹음이 성립하지 않는다.
   → v80에서 hls.js 스트림 바이트 캡처로 재설계.
3. 예약 녹음(07-16 추가)은 대기 선국과 동기 재귀가 만나 **UI 전체 프리징**
   (실측 1.5초에 selectStation 233만 호출), 예약 ID 재사용 + 발화 기록 영구 보존으로
   **재시도할수록 영원히 침묵하는 무덤돌**을 만들었다.
4. 예약 칩이 뜨면 모바일 플레이어 바에서 제목이 한 글자로 붕괴했다.
5. CI는 크로미움 전용이라 WebKit 회귀를 원천적으로 볼 수 없었고,
   webkit 프로젝트 추가 후에는 브라우저 미설치로 **3연속 실패를 방치**했다(R4에서 수정).

### 평가 방법의 구조적 결함 (점수가 부풀려진 이유)
- 모의 스트림 + 크로미움 전용 + 매번 새 상태 + 오디오 신호 무단언 —
  이 네 가지 조합에서 "완성도·안정성" 점수는 **주 플랫폼에 대해 반증 불가능**했다.
- 위 "한계" 절이 정확히 이 맹점을 최우선 항목으로 적시했음에도,
  점수에는 반영하지 않았다. **캐비앳을 프로즈에 적고 숫자는 깎지 않는 것이
  과대평가의 실체다.** 이후 개발도 같은 방법론을 물려받아
  "검증 완료" 주장을 반복했다.

### R4에서 마련한 검증 체계 (v76–v81)
- Playwright **webkit 프로젝트** 상설(예약·테이프 흐름), CI에 webkit 설치 추가
- 녹음 블롭 **실데이터(>20KB) 단언** — 무음·빈 파일 회귀 차단
- 모킹 없는 실스트림 검증 스크립트(WebKit, RMS/캡처 바이트 계측)
- 화면 내 버전 칩·예약 엔진 진단 라인 — "기기가 무엇을 돌리는지"를 눈으로 확인

**현재 점수는 산정 보류.** 사용자 실기기에서 예약 녹음 성공이 확인되기 전에는
숫자를 다시 매기지 않는다 — 같은 실수를 반복하지 않기 위한 원칙이다.
