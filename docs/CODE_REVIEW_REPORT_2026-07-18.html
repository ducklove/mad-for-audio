03 / Findings

우선순위별 핵심 발견

확정 결함 미확정 호환성 위험 구조적 부채

서비스워커가 루트·타 프로젝트 캐시를 손상시킬 수 있음

P1

활성화 과정에서 현재 캐시를 제외한 모든 origin 캐시를 삭제한다. GitHub Pages처럼 같은 origin에 여러 프로젝트가 있으면 다른 앱의 Cache Storage까지 영향을 받는다. 또한 모든 navigation 응답을 "./" 키에 기록해 설명서나 위젯 방문이 오프라인 루트 셸을 덮어쓸 수 있다.

  • fm-radio- 접두사 캐시만 정리한다.
  • 루트 navigation과 개별 페이지 캐시 키를 분리한다.
  • 루트·manual·widget 오프라인 전환 테스트를 추가한다.
sw.js:48–52 sw.js:78–91

사용자 테이프 라벨이 영속 SVG DOM 주입 경계가 됨

P1

직접 입력하거나 가져온 파일명에서 만든 테이프 라벨은 localStorage에 유지된다. 이후 랙 선반 SVG를 다시 그릴 때 라벨을 이스케이프 없이 innerHTML에 삽입한다. 로컬 사용자 입력이지만 Electron·PWA 문맥에서 지속 실행될 수 있는 명확한 DOM 주입 지점이다.

  • SVG text 노드를 생성하고 textContent를 사용한다.
  • HTML 문자열 렌더러와 사용자·외부 데이터의 경계를 정적 검사한다.
  • 장기적으로 인라인 이벤트를 제거하고 CSP를 적용한다.
deck.js:53–59 deck.js:901 deck.js:1172–1178 deck.js:1393–1395

실제 오디오와 재생 상태가 불일치할 수 있음

P1

플레이어를 붙인 직후 실제 playing 이벤트보다 먼저 isPlaying = true를 설정한다. HLS 미지원 콜백은 동기적으로 오류를 기록하지만 호출부가 곧바로 재생 중 상태로 덮어쓴다. 네이티브 HLS와 일반 파일의 오류 이벤트도 동일한 상태 모델로 수렴하지 않는다.

  • PlaybackController가 source, phase, requestId를 단독 소유한다.
  • UI·미디어 세션·램프는 상태 구독자로 제한한다.
  • 재생 성공은 실제 playing 이벤트에서만 확정한다.
player-core.js:57–67 app.js:3634–3674 app.js:3726–3729 app.js:2847–2856

예약 녹음 프리버퍼가 세션을 넘어 오염되고 장시간 메모리를 점유함

P1

rolling buffer는 새 청크를 계속 보관하지만 재튜닝 시 init과 MIME만 초기화된다. 짧은 간격의 연속 예약에서는 이전 채널 조각이 다음 녹음 앞에 섞일 수 있고, 기존 rolling 데이터가 고착 감지를 무력화한다. 수동·예약 녹음 모두 전체 데이터를 메모리에 모은 뒤 Blob으로 만들어 장시간 녹음에서 메모리 급증 가능성도 있다.

  • 예약·재튜닝별 session/generation ID를 도입한다.
  • 취소·완료·재튜닝 때 이전 이벤트와 rolling buffer를 폐기한다.
  • 청크를 IndexedDB 또는 OPFS에 점진 저장한다.
app.js:5282–5305 app.js:5308–5329 app.js:5334–5348 app.js:5391–5393

B면 삭제 누락과 저장 실패 은폐가 데이터 신뢰성을 낮춤

P1

녹음 삭제는 각 테이프의 segments만 정리하고 segmentsB는 남긴다. Blob URL과 IndexedDB 레코드가 삭제된 후 B면에는 죽은 트랙이 남는다. 또한 영속 저장 실패 시 null을 반환하지만 호출부가 성공 메시지와 세션 목록을 그대로 표시할 수 있다.

  • 양면 삭제·덮어쓰기·테이프 제거를 TapeRepository로 모은다.
  • 저장 API는 { ok, id, reason }을 반환하고 실패를 사용자에게 알린다.
  • quota 확인과 즉시 다운로드 폴백을 제공한다.
app.js:4198–4207 store.js:58–73 app.js:4022–4028

상시 RAF와 프레임별 DOM 조회가 유휴 성능을 소모함

P2

초기화 직후 tunerLoop()가 계속 RAF를 예약하고 508줄의 ttFrame()을 호출한다. 정적 계측상 프레임 본문에는 getElementById 29회, querySelectorAll 10회 이상, style 쓰기 27회, setAttribute 19회가 있다. 트레이는 background throttling을 꺼 두어 숨김 상태에서도 비용이 이어질 가능성이 높다.

  • 마운트 시 DOM 참조를 캐시하고 dirty flag로 변경분만 반영한다.
  • 재생·릴·웜업 등 활성 서브루프만 RAF를 사용한다.
  • 숨김·bar/simple 모드·reduced-motion에서 정지 또는 저주기 갱신한다.
app.js:404–459 app.js:2835–3342 app.js:4564 tray/main.js:105–112

Electron 런타임·빌드 체인과 보안 경계가 오래됨

P1

설치된 Electron 33.4.11과 electron-builder 25.1.8을 대상으로 한 npm audit에서 high 10건이 확인됐다. 원격 랙과 로컬 위젯을 한 창에서 다루면서 webSecurity: false도 설정되어 있다. 외부 URL 열기와 postMessage 수신 역시 허용 범위가 넓다.

  • Electron·builder를 지원되는 최신 메이저로 올리고 패키징 회귀를 검증한다.
  • webSecurity: true로 복원하고 스트림 CORS는 전용 계층에서 처리한다.
  • navigation, 외부 프로토콜, message source/origin을 명시적으로 제한한다.
tray/package.json tray/main.js:93–133 app.js:4567–4622 widget.html:1219–1285

모달·탭·피커의 ARIA 표기와 실제 키보드 동작이 불일치함

P2

여러 오버레이가 aria-modal="true"를 선언하지만 포커스 이동, 포커스 트랩, 호출자 복귀, 배경 inert가 없다. 편성표는 tablist만 있고 tab 역할, aria-selected, 방향키 탐색이 빠져 있다. 기존 axe 검사는 닫힌 모달 중심이라 이 상호작용을 잡지 못한다.

  • 공통 DialogController 또는 네이티브 dialog를 도입한다.
  • 피커의 active 상태를 aria-pressed와 동기화한다.
  • 열린 모달의 Tab 순환·ESC·포커스 복귀 E2E를 추가한다.
index.html:150–260 app.js:3353–3371 app.js:4695–4734 deck.js:1098–1108

대형 전역 스킨 레지스트리와 CSS cascade가 모델 추가 비용을 키움

P3

튜너·앰프·데크·턴테이블·타이머·EQ가 서로 다른 렌더 계약을 사용한다. component-skins가 앞서 로드된 전역 레지스트리를 뒤에서 변이하고, 모델 전환 때 전체 SVG 문자열을 재파싱한 뒤 이벤트를 다시 배선한다. CSS에도 후기 규칙이 이전 규칙을 덮는 구간과 퇴역 코드가 남아 있다.

  • render / mount / update / dispose / controls / audioProfile 계약을 통일한다.
  • 시각 스킨과 DSP 프로필을 분리하고 모델별 지연 로딩을 적용한다.
  • CSS를 tokens, base, rack, dialogs, pages 계층으로 분리한다.

app.js의 순수 코어·트레이 프로토콜과 CSS 계층은 물리 분리를 시작했지만, 화면·녹음 orchestration의 추가 ES module 분해와 모델별 지연 로딩은 별도 호환 전환이 필요하다. 다음 안전 단계는 기존 DB v1을 변경하지 않고 RecordingRepository의 open/read/status/removeMany 경계를 완성하는 것이다. 그 뒤 메모리 writer와 OPFS/IndexedDB 점진 chunk 저장을 순차 도입해야 기존 사용자 녹음을 보존할 수 있다. macOS Safari 실기·실제 방송 CORS·장시간 트레이 실행도 배포 전 최종 확인 대상으로 남긴다.

08 / Conclusion

최종 제안

가장 중요한 판단은 “파일 분할 자체가 목표가 아니다”라는 점이다. 현재 코드의 주된 위험은 복사·붙여넣기보다 여러 파일이 같은 재생·녹음·테이프 상태를 직접 변경하는 데서 생긴다. 따라서 서비스워커와 데이터 결함을 먼저 수정하고, PlaybackController, BackgroundRecorder, TapeRepository 순으로 상태 소유권을 옮긴 뒤 app.js와 스킨 계층을 분리해야 한다.

이 순서를 지키면 바닐라 JavaScript와 정적 배포라는 프로젝트의 장점, 그리고 현재의 강한 시각적 정체성을 유지하면서도 새 모델·새 소스·새 플랫폼을 추가할 때의 회귀 범위를 크게 줄일 수 있다.
